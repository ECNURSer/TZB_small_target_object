# YOLO26m 72 图过拟合测试

## 目的

该测试只回答一个问题：YOLO26m OBB 的模型容量、标签转换和训练链路是否能记住少量航拍小目标。train 和 val 故意使用同一组图像，因此结果不表示泛化能力，不能与正式 fold 0 val/test 分数比较。

## 数据和参数

- 从 fold 0 train 使用 seed 42 固定抽取 72 张图。
- 共 3443 个 OBB，10 类全部覆盖，单图最多 186 个目标。
- YOLO26m 官方 OBB 预训练权重。
- `imgsz=1024`、`batch=8`、GPU 1、AdamW、AMP。
- 恢复官方 BCE：`focal_gamma=0、cls_pw=0`。
- 所有增强关闭：`mosaic/mixup/cutmix/hsv/degrees/translate/scale/flips=0`。
- 训练到第 51 个完整记录后手动停止；实际评估使用可正常加载的 `best.pt`。

数据构建和训练命令：

```bash
python tools/overfit72.py --device 1
```

脚本默认 60 epoch，该数据在约 50 epoch 已达到过拟合判定标准。

## 结果

`best.pt` 在同一 72 图集合的 Ultralytics 复评：

| Precision | Recall | mAP50 | mAP50-95 |
|---:|---:|---:|---:|
| 0.9020 | 0.9743 | 0.9775 | 0.8083 |

本地比赛评分器在 IoU 0.3 下精确搜索 conf：

| conf | TP | FP | FN | Precision | Recall | F1@0.3 |
|---:|---:|---:|---:|---:|---:|---:|
| 0.51865 | 3346 | 64 | 97 | 0.9812 | 0.9718 | 0.9765 |

结论：YOLO26m 能够过拟合这组航拍小目标 OBB。正式数据上的错误更应从泛化、类别不平衡、输入分辨率、增强和标注噪声方面分析，不是模型完全没有小目标 OBB 表达能力。

## degrees=180 是否合理

过拟合测试必须使用 `degrees=0`。若设为 180，每轮看到的是随机旋转新样本，测试就同时混入了增强鲁棒性，无法单独判断模型是否能记住原始标注。

对正式航拍车辆训练，若相机航向和目标方向没有固定语义，`degrees=180` 在语义上是合理的，可以覆盖全方位方向。新的 m 全量单模型配置已采用 180 度，但它不一定比历史基线的 15 度更好：非方形图像旋转会引入边界填充和目标裁切，大角度增强也会提高学习难度。后续仍应在 fold 0 val 上保持其他参数不变，比较 `degrees=15、90、180`，并为每组重新搜索比赛 conf。
